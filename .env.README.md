# Environment Variables for Web App

## Quick Start

```bash
# 1. Use root .env.example as reference
cp ../../.env.example ../../.env.local

# 2. Initialize Convex (auto-generates .env.local here)
bunx convex dev
```

## How It Works

### Root Level (.env.example)
- **Location:** `/drawink/.env.example`
- **Purpose:** Documentation for ALL environment variables
- **Usage:** Reference for what variables are needed

### App Level (.env.local)
- **Location:** `/drawink/apps/web/.env.local` (this directory)
- **Purpose:** Auto-generated by Convex CLI
- **Contains:** `VITE_CONVEX_URL` and `CONVEX_DEPLOYMENT`
- **Important:** This file is created automatically when you run `bunx convex dev`

### Environment Variable Loading

Vite loads environment variables in this order:
1. `.env.local` (this directory) - **Convex auto-generated**
2. `.env` (this directory) - Manual overrides
3. `.env.production.local` (for production builds)

## What Goes Where?

### Auto-Generated (Don't Edit!)
```bash
# apps/web/.env.local (created by Convex CLI)
CONVEX_DEPLOYMENT=dev:compassionate-ibis-308
VITE_CONVEX_URL=https://compassionate-ibis-308.convex.cloud
```

### Manually Added (Create .env in this directory)
```bash
# apps/web/.env
VITE_CLERK_PUBLISHABLE_KEY=pk_test_xxx
VITE_APP_FIREBASE_CONFIG={"apiKey":"xxx",...}
```

## Why Two Locations?

1. **Root `.env.example`** - Central documentation for developers
2. **App `.env.local`** - Convex CLI requirement (must be in same directory as convex/ folder)

This is standard practice in monorepos where:
- Root = Documentation
- App = Runtime configuration

## Adding New Variables

1. Add documentation to root `.env.example`
2. Add actual value to `apps/web/.env` (NOT .env.local, Convex owns that)
3. Commit `.env.example`, ignore `.env` and `.env.local`
